分布式一致性算法

分布式一致性算法包括Paxos和Raft、ZAB协议

Paxos算法

​ Paxos算法是基于消息传递且具有高度容错特性的一致性算法，是目前公认的解决分布式一致性问题最有效的算法之一。来自Stanford的新的分布式协议研究称为Raft，它是一个为真实世界应用建立的协议，主要注重协议的落地性和可理解性。

​ Paxos算法分为Basic Paxos，Multi Paxos、Fast Paxos。

Basic Paxos

角色

在Paxos算法中，有三种角色：

  • Proposer：提议者，只要Proposer发的提案被半数以上Acceptor接受，Proposer就认为该提案通过了。否则就把提案号加1，尝试再次发送提案。
  • Acceptor：投票者，只要Acceptor接受了某个提案，Acceptor就认为该提案里的value被选定了。
  • Learners：学习者/记录员，Acceptor告诉Learner哪个value被选定，Learner就认为那个value被选定。

​ 在Basic Paxos算法中，一个进程可能同时充当多种角色。比如一个进程可能既是Proposer又是Acceptor又是Learner。

请求与响应

Basic Paxos中包括两种请求和两种响应，我们按照顺序来介绍。

  • Prepare请求：提案号N同步请求
  • Promise响应：提案号N同步响应
  • Accept请求：提案N，V同步请求
  • Accepted响应：提案N，V同步响应

流程

Paxos算法简单分为四种信号状态

  1. Prepare:Proposer向所有已知的Accepter提出一个Prepare请求，请求包括提案号N，这个提案号大于Proposer之前提过的提案号。
  2. Promise：如果N大于Accepter之前所响应过的提案号，则接受N这个提案号，发送Promise响应，否则拒绝。
  3. Accept：Proposer所收到Accepter的Promise响应达到了多数（一般为半数以上），Proposer则会发送Accept请求，该请求包括提案号N与提案内容V。
  4. Accepted_1：只要该Acceptor没有对编号大于N的Prepare请求做出过响应，它就接受该提案，发送Accepted响应，否则就拒绝。
  5. Accepted_2：Proposer所接收的该事件Accepted响应达到了多数，则提案通过，否则，将提案号+1再次重复第一步步骤；并将表决结果告知Learner。

Paxos算法分为两个阶段。分为提案号N同步，和提案（N，V）同步，具体如下：

  • 阶段一：选出提案编号最大的Proposer（提案号N同步）

    (a) Proposer选择一个Proposal编号N，然后向半数以上的Acceptor发送编号为N的Prepare请求。

    (b) 如果一个Acceptor收到一个编号为N的Prepare请求，且N大于该Acceptor已经响应过的所有Prepare请求的编号，那么它就会将它已经接受过的编号最大的提案（如果有的话）作为响应反馈给Proposer，同时该Acceptor承诺不再接受任何编号小于N的提案。

    ​ 如果N小于等于该Acceptor之前批准过的任意Prepare请求的编号，则拒绝该编号N的Prepare请求。

  • 阶段二：尝试与某一Proposer达成一致（提案N，V同步）

    (a) 如果Proposer收到半数以上Acceptor对其发出的编号为N的Proposal请求的响应，那么它就会发送一个针对[N,V]提案的Accept请求给半数以上的Acceptor。

    ​ 注意：V就是收到的响应中编号最大的提案的value，如果响应中不包含任何提案，那么V就由Proposer自己决定。

    (b) 如果Acceptor收到一个针对编号为N的提案的Accept请求，

    ​ 只要该Acceptor没有对编号大于N的Prepare请求做出过响应，它就接受该提案，

    ​ 只要该Acceptor对编号大于N的Prepare请求做出过响应，它就拒绝该提案，

缺陷

​ 实现复杂

​ 多个Proposer的实现比较复杂，节点之间的通信量大。

​ 效率低

​ 需要进行两轮RPC调用，共计四次通信才能确定一个提案，效率较低。

​ 活锁

​ Basic Paxos可能出现活锁，如果多个Proposer竞争提案号，就会出现长时间没有提案能够接受的情况。下图为活锁示例。解决方案为提案失败则随机时间重发，不过这样的实现方式，效率依旧较低。

Multi Paxos

角色

  • Leader：唯一的Proposer（强Leader状态），只要Leader发的提案被半数以上Acceptor接受，Leader就认为该提案通过了。
  • Acceptor：只要Acceptor接受了某个提案，Acceptor就认为该提案里的value被选定了。
  • Learners：Acceptor告诉Learner哪个value被选定，Learner就认为那个value被选定。

流程

​ Multi Paxos和Basic Paxos 状态相似，有四个信号状态。不过之前的提案号N同步的状态只有在开始时使用（类似于选举，但不完全是，后面会有提到），而后面集群中的绝大部分时间只会有Accept和Accepted两种情况。

  1. Proposer（leader）向所有已知的Accepter提出一个Prepare请求，请求包括提案号N，这个提案号大于Proposer之前提过的提案号。

  2. Promise：如果N大于Accepter之前所响应过的提案号，则接受N这个提案号，发送Promise响应，否则拒绝。其中Promise包括了N和新的事件号I（I可以理解为约定的新提案号，后面流程的N则不会改变，N相当于Acceptor识别这个Leader的标识）

  3. Accept：Proposer所收到的Promise接受响应达到了多数（一般为半数以上），Proposer则会发送Accept请求，该请求包括提案号N与事件号I和提案内容V（N，I，V）

  4. Accepted_1：只要该Acceptor没有对提案号为N且事件编号大于I的Accept请求做出过响应，它就接受该提案，发送Accepted响应，否则就拒绝。

  5. Accepted_2：

    ​ 1）如果Acceptor无返回值，则自由决定一个Value；

    ​ 2）如果有返回值，则选择事件号I最大的一个；

    ​ Proposer所接收的该事件Accepted响应达到了多数，则提案通过，否则，将提案号+1再次重复第一步步骤；Acceptor把表决结果发送到Learner。

接下来的操作则是通过Leader的一次Accept请求即可完成一次提案，比起Basic Paxos两轮确定更有效率。

确定Leader

​ 之前提到在第一次Proposer和Acceptor交互时要确定Leader，那是如何确定Leader的呢，个人理解是哪个结点提交提案越快越频繁则越有可能成为Leader。

​ 需要注意的一点是，Multi Paxos算法是允许并行提交的，意思是Multi Paxos并不是完全的禁止其他节点发送Prepare请求。只是Multi Paxos在设计时有一条规则：

​ 结点收到其他节点的Accept，则进行一段时间的拒绝提交提案（暂时放弃Proposer身份）。

​ Leader的作用是希望大部分时间都只有一个节点在提交，这样才能最大发挥Mulit-Paxos的优化效果。

​ 所以最好在Leader可用的情况下，不要打破这种状态，各个节点都想着不要去打破这种连续的Accept状态。而当有一个节点（即可能的Leader）在连续的Accept，那么其他节点必然持续不断的拒绝提交提案。这个Leader就这样无形的被产生出来了。

Fast Paxos

注意这里的Classic Paxos和Basic与Multi都不太相同。

角色

  • Client/Proposer/Learner：负责提案并执行提案
  • Coordinator：Proposer协调者，可为多个，Client通过Coordinator进行提案
  • Leader：在众多的Coordinator中指定一个作为Leader
  • Acceptor：负责对Proposal进行投票表决

Any消息

​ Acceptor在接收到Any消息后可做一些开始Fast Round的初始化工作，等待Proposer提交真正的Value。Any消息的意思是Acceptor可以做任意的处理。 个人理解，Any消息类似于一种信号，由Leader发送给Acceptor后，Acceptor乐观地接收其他Proposer的提案，而如果发生冲突（即收到的提案号或者内容冲突），则需要Leader再次发送最新的提案编号来同步，修复冲突。

​ 所以说Fast Paxos是基于乐观锁的思想，Acceptor认为冲突是一件不经常发生的事件。这样的架构在并发较大、事件较多时效率较低。

流程

让我们来回忆一下Classic Paxos中一个提案是如何确定的（排除第一次连接）

  • Accept：Leader提交proposal到Acceptor
  • Accepted_1：Acceptor回应已经参与投票的最大Proposer编号和选择的Value
  • Accepted_2：Leader收集Acceptor的接受返回值
    • 1）如果Acceptor无返回值，则自行决定一个
    • 2）如果有返回值，则选择Proposer编号最大的一个
    • Proposer所接收的该事件Accepted响应达到了多数，则提案通过，否则，将提案号+1再次重复第一步步骤；Acceptor把表决结果发送到Learner。

Classic Paxos中，从每次Proposer提案到决议被学习，需要三个通信步骤：

​ Proposer—–Leader—–Acceptor—–Learner

​ 如果Leader可以自行决定一个Value，则可以让Proposer提交这个Value，Leader自己则退出通信过程。只要之后的过程运行正常，Leader始终不参与通信，一直有Proposer直接提交Value到Acceptor，从而把Classic Paxos的三阶段通信减少为两阶段，这便是Fast Paxos的由来。

  • Accept：Leader提交proposal到Acceptor
  • Accepted_1：Acceptor回应已经参与投票的最大Proposer编号和选择的Value
  • Accepted_2：Leader收集Acceptor的接受返回值
    • 1）如果Acceptor无返回值，则发送一个Any消息给Acceptor，之后Acceptor便等待Proposer提交Value。
    • 2）如果有返回值，则选择Proposer编号最大的一个
    • Proposer所接收的该事件Accepted响应达到了多数，则提案通过，否则，将提案号+1再次重复第一步步骤；Acceptor把表决结果发送到Learner（包括Leader）

形式消息仅需两个通信步骤

​ Proposer—–Acceptor—–Learner

无冲突流程

​ Fast Paxos generalizes Basic Paxos to reduce end-to-end message delays. In Basic Paxos, the message delay from client request to learning is 3 message delays. Fast Paxos allows 2 message delays, but requires that

​ (1) the system be comprised by 3f+ 1 acceptors to tolerate up to f faults (instead of the classic 2f+1), and

​ (2) the Client to send its request to multiple destinations.

​ Fast Paxos概括了Basic Paxos以减少端到端消息延迟。在Basic Paxos中，从客户端请求到学习的消息延迟是3个消息延迟。 Fast Paxos允许2个消息延迟，但要求

​ （1）系统由3f + 1个接收器组成以容忍多达f个故障（而不是经典的2f + 1），以及

​ （2）客户端将其请求发送到多个目的地。

冲突流程

Multi和Fast的区别

​ 两种算法从形式上比较相似，都只是由一轮RPC则可以确定一次提案。而两者之间的不同有：

  1. 提案的提交者：Multi形式上只有一个Leader来提交提案，而Fast则是多个Proposer提交提案。
  2. 两者Leader作用不同：Multi的Leader用于提交提案，即唯一的Proposer；而Fast的Leader主要用于解决冲突，平时的角色和Learner类似。
  3. Fast的客户端即作为了Proposer的角色，发送请求时与Proposer相同，发送给多个Acceptor。

Raft算法

​ Raft是一种强领导算法。详细示例可见Raft算法动画演示 与Raft算法演示网站

角色

  • Leader：领导者，请求都通过Leader来发送。向Follower发送HeartBeat心跳检测，HeartBeat可能携带请求。
  • Follower：跟随者，集群正常的情况下，则为记录员，每个Follower都有随机超时时间（每次收到HeartBeat随机重置），一旦在超时时间内没有收到来自与Leader的HeartBeat，则Follower升级为Candidate，该节点就会参与竞选。
  • Candidate：候选者，一旦Timeout超时，Follower升级为Candidate，则可以竞选Leader。如果在选举时，发现了新的Leader，则会变为Follower。

选举Leader

​ 每个Follower都有一个Timeout，超时时间，如果在超时时间内，没有Leader发送的HeartBeat，那Follower则会变为Candidate发送选举请求，一旦超过半数的Follower同意，则新的Leader就选举出来了。

  1. 第一种情况是初始情况，所有的Follower都升级为Candidate竞选Leader，所有Candidate竞选的Term都相同，则谁先获得半数同意，谁就是Leader。
  2. 如果有一个Follower在任期N内超时，则自己升级为Candidate（称为X节点），X选择N+1任期，竞选Leader。这里有一个规则，如果一个Candidate或者Leader获得一个大于自身任期的RPC（即新的请求），那它们就直接更新自己的任期（即保持最新），Candidate或者Leader自动降级为Follower；如果X的RPC’s Term小于Candidate或者Leader，Candidate或者Leader则拒绝该请求（注）。
    • 注：Candidate或者Leader拒绝RPC竞选请求时，需要检查源CandidateX的日志同步状态，根据Term选择是否要将源CandidateX进行日志更新同步，因为如果CandidateX的Term小于当前集群里的Term，大概率是之前X节点宕机了，而X恢复时又触发了超时，这个时候X是很有可能缺少一些日志的。所以需要Leader来帮助X节点同步。

日志复制（信息同步）

​ 日志复制（Log Replication）主要作用是用于保证节点的一致性，这阶段所做的操作也是为了保证一致性与高可用性；

​ 当Leader选举出来后便开始负责客户端的请求，所有事务（更新操作）请求都必须先经过Leader处理，这些事务请求或说成命令也就是这里说的日志，我们都知道要保证节点的一致性就要保证每个节点都按顺序执行相同的操作序列。流程如下：

  1. Leader得到Client的日志（事务请求），Leader追加到本地日志中，并通过HeartBeat向Follower发送日志复制同步请求与日志追加内容（也可以理解为命令，例如 SET 5），
  2. Follower接收到日志后记录日志然后向Leader发送ACK，注意这时Follower只是发送了ACK，并没有写入本地磁盘中。
  3. 当Leader收到大多数（n/2+1）Follower的ACK信息后将该日志设置为已提交并追加到本地磁盘中，并通知客户端并在下个heartbeat中Leader将通知所有的Follower将该日志存储在自己的本地磁盘中，并把自己事务号加1，表示完成了一次事务。
  4. Follower接收到日志写入的命令，将日志写入本地磁盘。

安全性

​ 安全性是用于保证每个节点都执行相同序列并保持阶段一致性的安全机制，如当某个Follower在当前Leader commit Log时变得不可用了，稍后可能该Follower又会倍选举为Leader，这时新Leader可能会用新的Log覆盖先前已committed的Log，这就是导致节点执行不同序列Safety就是用于保证选举出来的Leader一定包含先前 commited Log的机制；

  • 选举安全性（Election Safety）
    • 每个Term只能选举出一个Leader
  • Leader完整性（Leader Completeness）
    • 这里所说的完整性是指Leader日志的完整性，当Log在Term1被Commit后，那么以后Term2、Term3…等的Leader必须包含该Log；Raft在选举阶段就使用Term的判断用于保证完整性：当请求投票的该Candidate的Term较大或Term相同Index更大则投票，否则拒绝该请求；因为如果新竞选的节点X的Term小于当前Term时，则X是需要Leader来进行日志同步的，需要将X之前少的日志都补上。X也就谈不上竞选了。

ZAB（Zookeeper 原子广播协议）

​ Zookeeper 是一个为分布式应用提供高效且可靠的分布式协调服务。在解决分布式一致性方面，Zookeeper 并没有使用 Paxos ，而是采用了 ZAB 协议。

​ ZAB 协议是为分布式协调服务 Zookeeper 专门设计的一种支持 崩溃恢复 和 原子广播 协议。

节点状态

节点可能的状态有以下四种

  • following：当前节点是跟随者，服从 leader 节点的命令
  • leading：当前节点是 leader，负责协调事务
  • election/looking：节点处于选举状态，或者是新加入节点，该节点处于恢复模式
  • observing： 观察状态，不参与选举。相当于记录员

节点中包含以下属性：

  • history：当前节点接收到事务提议的 log
  • acceptedEpoch：follower 已经接受的 leader 更改年号的 NEWEPOCH 提议
  • currentEpoch：当前所处的年代
  • lastZxid：history 中最近接收到的提议的 zxid （最大的）

ZXID

​ zxid是一个64位的数字，前32位是Epoch纪元号，表达现在是哪个Leader在领导，每次一个leader被选出来，它都会有一个新的epoch，类似与Raft中的Term；后32位是事件号，表示现在的同步流程。

​ 低 32 位是消息计数器，每接收到一条消息这个值 +1，新 leader 选举后这个值重置为 0。这样设计的好处是旧的 leader 挂了后重启，它不会被选举为 leader，因为此时它的 zxid 肯定小于当前的新 leader。当旧的 leader 作为 follower 接入新的 leader 后，新的 leader 会让它将所有的拥有旧的 epoch 号的未被 COMMIT 的 proposal 清除。

崩溃恢复

​ 如果leader服务器发生崩溃，则zab协议要求zookeeper集群进行崩溃恢复和leader服务器选举。

​ ZAB协议崩溃恢复要求满足如下2个要求：

​ 1）确保已经被leader提交Commit的proposal必须最终被所有的follower服务器提交。

​ 2）确保丢弃已经被leader出的但是没有被提交Commit的proposal。

​ 其实以上的要求就要保证新选举出来的leader不能包含未提交的proposal，即新选举的leader必须都是已经提交了的proposal的follower服务器节点。同时，新选举的leader节点中含有最高的ZXID。这一切都由ZXID和选举逻辑完成的。原理如下：

  • ​ 确保Commit提交：因为选举逻辑就是选出ZXID最大的节点，一旦前任Leader的最后一次Commit有节点操作了，那该节点的ZXID必然最大（因为Commit后LastZXID更新了），也就必然是下一轮的Leader，那新Leader必然会与其他的Follower同步，将前任Leader最后一次Commit同步到所有节点。

  • ​ 丢弃未Commit的事务：而如果前任Leader最后一次操作为ACK但未Commit，那新Leader必然没有更新这一次的ZXID，又因为新Leader选举出来后，事务号会清零，那Follower就必然丢弃了前任Leader未Commit的操作。（此处有争议，具体实现可能不同，可能新Leader在检查所有Follower状态时会根据Follower中已ACK的数量来决定这次事务是否同步）。

​ 下面来介绍ZAB在崩溃恢复中的选举逻辑FastLeaderElection。

  1. 当leader崩溃或者leader失去大多数的follower，或者初始状态，这时候zk进入恢复模式，恢复模式需要重新选举出一个新的leader，让所有的server都恢复到一个正确的状态。Follower状态为Looking。
  2. 选举线程由当前节点发起选举的线程担任，功能是对投票结果进行统计，并选出leader。
  3. 选举线程首先向所有 Server 发起一次询问(包括自己)，获得当前所有节点的状态信息。（第一轮发起推荐）
  4. 选举线程收到回复后，验证是否是自己发起的询问(验证 zxid 是否一致)，然后获取对方 的 节点id(myid)，并存储到当前询问对象列表中，最后获取对方提议的 leader 相关信息 (节点id,zxid)，并将这些信息存储到当次选举的投票记录表中。（第一轮获得推荐）

  5. 收到所有 Server 回复以后，就计算出 id 最大的那个 Server，并将这个 Server 相关信息设 置成下一次要投票的节点（计算ID的规则如下）

    1. Epoch小的选举结果被忽略，重新投票。
    2. 统一Epoch后，事件号大的胜出，
    3. 事件号相同的情况下，节点 id 大的胜出

  6. 线程将当前 id 最大的 Server 设置为当前 Server 要推荐的 Leader，如果此时获胜的 Server 获得 n/2 + 1 的 Server 票数， 设置当前推荐的 leader 为获胜的 Server，将根据获胜的 Server 相关信息设置自己的状态，否则，继续这个过程，直到 leader 被选举出来。（第二轮确定Leader）（注）

注：如果在选举完成后一个新的或者是宕机恢复的节点 加入zookeeper服务中，它会在恢复模式（Looking）下启动，每个节点启动以后都询问其它节点它要投票给谁，直到发现leader，并和leader进行状态同步。待到同步结束，它也参与消息广播。

注：在zookeeper集群中新的leader选举成功之后，leader会将自身的提交的最大proposal的事物ZXID发送给其他的follower节点。follower节点会根据leader的消息和自己的lastZxid进行回退或者是数据同步操作。最终目的要保证集群中所有节点的数据副本保持一致。

消息广播

​ 类似一个 二阶段提交过程。对于客户端发送的写请求，全部由 Leader 接收，Leader 将请求封装成一个事务 提案，将其发送给所有 Follower ，然后，根据所有 Follower 的反馈，如果超过半数成功响应，则执行 commit 操作（先提交自己，再发送 commit 给所有 Follower）。其实这里的和Raft的日志同步非常相似。

  1. Leader得到Client的日志（事务请求），Leader追加到本地日志中，并通过HeartBeat向Follower发送日志复制同步请求与日志追加内容（也可以理解为命令，例如 SET 5），
  2. Follower接收到日志后记录日志然后向Leader发送ACK，注意这时Follower只是发送了ACK，并没有写入本地磁盘中。
  3. 当Leader收到大多数（n/2+1）Follower的ACK信息后，将该日志设置为已提交并追加到Leader本地磁盘中，并通知客户端并在下个heartbeat中Leader将通知所有的Follower将该日志存储在自己的本地磁盘中（即Commit操作），并把自己事务号加1，表示完成了一次事务。
  4. Follower接收到日志写入Commit的命令，将日志写入Follower本地磁盘。

ZAB和Raft的异同

ZAB和Raft有很多相似的地方，比如几乎相同的日志同步机制，几乎相同的Leader同步断连的Follower的思想，基本相同的Leader选择逻辑，都是靠Epoch(Term)和事务号来解决数据同步问题。不过它们还有一些不同之处。

  1. ZAB的心跳检查是由Follower发至Leader或者Follower，而Raft是由Leader发送至Follower。
  2. ZAB的选举模式是由一个结点统一记录投票情况，并有两轮确定，第一次是确定推荐，第二次确定Leader。而Raft由各个节点自己统计自己的得票情况，并且一旦Term和事务号满足大于条件，则直接获得成为Leader，只有一轮确定。
